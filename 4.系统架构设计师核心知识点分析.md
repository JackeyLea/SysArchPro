# 系统架构设计师核心知识点分析

## 一、核心架构设计概念

### 1.1 软件架构基础概念

#### 1.1.1 架构定义与4+1视图
- **架构定义**：软件架构是系统的一组结构，包含软件元素、元素的外部可见属性、元素之间的关系
- **软考中的4+1视图**：
  - 逻辑视图：面向对象设计，描述功能需求、对象模型
  - 开发视图（实现视图）：关注软件开发环境、描述软件静态组织结构
  - 进程视图：关注并发和同步
  - 物理视图（部署视图）：关注软件到硬件的映射
  - 场景视图（用例视图）：用例或场景，整合其他视图，描述系统如何满足用户需求

#### 1.1.2 基于架构的软件开发方法(ABSD)

| 考点 | 重要程度 | 说明 |
|------|----------|------|
| ABSD的**六个子过程** | 高频 | 需求、设计、文档化、复审、实现、演化 |
| ABSD的三个基础 | 高频 | 功能分解、架构风格、软件模板 |
| 视角与视图 | 高频 | 描述软件架构的核心方法 |
| 用例与质量场景 | 高频 | 描述功能需求和质量需求 |
| 架构文档化输出 | 中频 | 架构规格说明书、质量设计说明书 |

## 二、系统质量属性与架构评估（超级重点★★★★★）

### 2.1 九大必考质量属性

| 质量属性 | 描述 | 对策 |
|---------|------|------|
| **性能** | 系统响应能力，处理事务所需时间 | 优先级队列、增加计算资源、引入并发机制、资源调度 |
| **可靠性** | 在错误情况下维持功能的能力 | 冗余设计、错误检测与恢复、异常处理、备份恢复 |
| **可用性** | 系统正常运行的时间比例 | 心跳机制、冗余系统、快速恢复、故障转移 |
| **安全性** | 阻止非授权访问的能力 | 用户认证、授权管理、数据加密、防火墙、审计日志 |
| **可修改性** | 快速高性价比变更系统的能力 | 模块化设计、接口标准化、抽象封装、配置管理 |
| **互操作性** | 与其他系统交互的能力 | 标准协议(RESTful/SOAP)、数据格式转换、中间件 |
| **可移植性** | 适应不同运行环境的能力 | 标准API、抽象底层差异、虚拟机/容器化 |
| **可测试性** | 软件能被有效测试的难易程度 | 分层模块化、提供测试接口 |
| **易用性** | 系统让用户好学好操作的程度 | 统一交互规范、信息架构清晰、即时反馈 |

### 2.2 质量属性场景描述（六要素）

**质量属性场景 = 刺激源 + 刺激 + 环境 + 制品 + 响应 + 响应度量**

#### 各质量属性场景要素：

**可用性场景：**
- 刺激源：系统内部、系统外部
- 刺激：疏忽、错误、崩溃、时间
- 环境：正常模式到降级模式
- 制品：系统处理器、通信信道、持久存储器、进程
- 响应：检测事件、记录事件、通知事件、隔离故障、等待恢复
- 响应度量：故障修复时间

**性能场景：**
- 刺激源：用户请求、其他系统触发
- 刺激：事件到达
- 环境：正常模式到超载模式
- 制品：系统
- 响应：处理刺激、改变服务级别
- 响应度量：等待时间、期限、吞吐量、抖动、缺失率

**可修改性场景：**
- 刺激源：最终用户、开发人员、系统管理员
- 刺激：改变功能、扩容等
- 环境：系统设计时、编译时、构建时、运行时
- 制品：系统用户界面、平台、环境
- 响应：修改且不会影响其他功能
- 响应度量：成本、努力、资金；对其他功能的影响

### 2.3 架构评估方法

#### 2.3.1 基于场景的评估方法
| 方法 | 全称 | 主要关注点 |
|-----|------|-----------|
| **SAAM** | 基于场景的架构分析方法 | 场景分析、可修改性，协调风险承担者达成共识 |
| **ATAM** | 架构权衡分析方法 | 性能、安全性、可修改性、可用性的评价和折中 质量属性权衡、效用树 |
| **CBAM** | | 成本效益分析、ROI评估 |

#### 2.3.2 ATAM评估四阶段
1. **演示阶段**：介绍ATAM流程、业务目标、待评估架构
2. **调查和分析**：确定架构设计方法；生成质量属性效用树；分析架构与质量属性匹配度
3. **测试阶段**：头脑风暴提出候选场景；筛选优先级场景验证架构支持能力
4. **报告阶段**：汇总评估结果（风险、非风险、敏感点、权衡点）

#### 2.3.3 关键概念辨析
| 概念 | 定义 | 举例 |
|-----|------|------|
| **敏感点** | 一个或多个构件的特性 | 查询请求处理时间要求影响数据传输协议设计（影响单一属性） |
| **权衡点** | 影响多个质量属性的特性 | 加密级别选择影响安全性和性能（影响多个属性） |
| **风险点** | 可能导致架构问题的特性 | 如果业务逻辑未达成共识，可能影响可修改性 |
| **非风险点** | 不会造成负面影响的特性 | 选择成熟消息队列中间件不会带来明显风险 |

## 三、云原生架构设计（超级重点★★★★★）

### 3.1 云原生架构原则
| 原则 | 描述 |
|-----|------|
| **服务化原则** | 项目规模大时拆分服务，提升复用性与治理能力 |
| **弹性原则** | 系统自动伸缩，适配业务需求 |
| **可观测性原则** | 通过日志、链路跟踪等掌握系统状态 |
| **韧性原则** | 抵御软硬件异常，保障系统稳定 |
| **全流程自动化原则** | 自动化交付提升效率、降低风险 |
| **零信任原则** | 基于身份访问控制，替代传统网络边界安全 |
| **持续演进原则** | 架构适应技术与业务变化，渐进式升级 |

### 3.2 云原生架构模式
| 模式 | 核心定义 |
|-----|---------|
| **服务化架构** | 按业务域拆分应用为独立服务，通过API交互 |
| **Mesh化架构** | 将中间件能力从业务进程剥离，由Sidecar统一处理 |
| **Serverless模式** | 无需管理服务器，云平台自动完成部署、运行、伸缩 |
| **存储计算分离** | 计算层与存储层解耦，计算节点保持无状态 |
| **分布式事务** | XA/TCC/SAGA/消息最终一致性等模式 |
| **可观测架构** | Logging/Tracing/Metrics三大核心能力 |
| **事件驱动架构(EDA)** | 系统以事件为核心交互，事件作为独立数据单元 |

### 3.3 容器与虚拟机对比（重点★★★★★）
| 对比维度 | 容器 | 虚拟机 |
|---------|------|--------|
| **运行原理** | 基于OS层面虚拟化，共享宿主机内核 | 硬件虚拟化，模拟完整硬件环境 |
| **资源隔离** | cgroups资源限制+namespace命名空间隔离 | 硬件虚拟化实现完全资源隔离 |
| **启动速度** | 秒级 | 分钟级 |
| **资源占用** | 占用少，共享内核 | 占用多，独立OS和硬件模拟 |
| **隔离性** | 相对较弱，共享内核 | 隔离性强，相互独立 |

## 四、面向服务架构SOA（超级重点★★★★★）

### 4.1 SOA核心概念
- **定义**：将应用程序的不同功能单元（服务）通过定义良好的接口和契约联系起来
- **特点**：接口采用中立方式定义，独立于硬件平台、操作系统和编程语言

### 4.2 SOA/微服务/单体架构对比

| 特性 | 微服务 | SOA | 单体架构 |
|-----|--------|-----|---------|
| **服务粒度** | 极为精细，单一业务功能 | 相对较大，多个业务功能 | 紧密耦合整体 |
| **通信方式** | HTTP RESTful API | HTTP/SOAP多种协议 | 内部方法调用 |
| **数据管理** | 每个服务独立数据库 | 共享数据库或ESB交换 | 单一数据库 |
| **架构风格** | 去中心化 | 可能包含中心化ESB | 集中式架构 |

### 4.3 服务注册模式核心协议
| 协议 | 概念 | 主要作用 |
|-----|------|---------|
| **UDDI** | 通用描述、发现和集成服务 | 统一发布和搜索Web服务信息 |
| **WSDL** | Web服务描述语言 | 定义服务功能、参数、绑定协议 |
| **SOAP** | 简单对象访问协议 | 定义消息格式、编码规则、RPC约定 |

### 4.4 ESB（企业服务总线）
- **功能**：位置透明性的消息路由和寻址、服务注册和命名管理
- **支持**：多种消息传递范型、多种传输协议、多种数据格式转换
- **附加**：日志和监控功能

### 4.5 微服务模式分类
1. **聚合器微服务**：将多个API组合成统一接口
2. **代理微服务**：客户端与服务间增加中间层
3. **链式微服务**：服务按顺序串联
4. **分支微服务**：按业务子系统拆分
5. **数据共享微服务**：统一数据访问接口
6. **异步消息微服务**：通过消息队列异步通信

## 五、分布式系统与数据库（重点★★★★★）

### 5.1 CAP理论
| 特性 | 描述 |
|-----|------|
| **一致性(C)** | 所有节点在同一时间看到相同数据 |
| **可用性(A)** | 每个请求都能获得响应，不保证数据最新 |
| **分区容错性(P)** | 系统在网络分区时仍能继续运行 |

**CAP定理**：分布式系统最多同时满足其中两项

### 5.2 BASE理论
- **基本可用(Basically Available)**
- **软状态(Soft State)**
- **最终一致性(Eventually Consistent)**

### 5.3 分布式事务解决方案
| 方案 | 特点 |
|-----|------|
| **2PC（两阶段提交）** | 刚性事务，强一致，性能差 |
| **TCC** | 柔性事务，Try/Confirm/Cancel，灵活但侵入性强 |
| **本地消息表** | 柔性事务，异步保证最终一致性 |
| **SAGA** | 长事务分解为本地事务+补偿操作 |

### 5.4 数据库高可用模式
| 模式 | 描述 |
|-----|------|
| **读写分离** | 主库写、从库读，提升读性能 |
| **主从复制** | 数据从主库同步到从库 |

## 六、大数据架构（重点★★★★★）

### 6.1 Lambda架构
| 层次 | 功能 |
|-----|------|
| **批处理层** | 存储主数据集，进行批量处理 |
| **速度层** | 处理实时数据流 |
| **服务层** | 合并批处理和实时结果，提供查询 |

**优缺点**：
- 优点：容错性好、数据一致性高
- 缺点：维护两套代码复杂、实时性延迟

### 6.2 Kappa架构
- **核心思想**：只保留速度层，用流处理统一处理所有数据
- **与Lambda区别**：无需维护批处理和流处理两套代码

### 6.3 Lambda vs Kappa
| 对比项 | Lambda | Kappa |
|-------|--------|-------|
| 数据处理 | 批处理+流处理 | 仅流处理 |
| 代码维护 | 两套代码 | 一套代码 |
| 复杂度 | 较高 | 较低 |
| 实时性 | 有延迟 | 更好 |

## 七、安全架构设计（超级重点★★★★★）

### 7.1 安全架构三大特性
| 特性 | 定义 | 典型例子 |
|-----|------|---------|
| **可用性** | 确保系统资源在需要时可正常访问 | 数据备份与恢复、负载均衡、双机热备 |
| **完整性** | 保护数据不被未经授权修改 | 哈希校验、数字签名、事务回滚 |
| **机密性** | 防止敏感信息被泄露 | 数据加密、HTTPS、访问控制列表 |

### 7.2 数据加密技术
| 对比项 | 对称加密 | 非对称加密 |
|-------|---------|-----------|
| **密钥** | 单一密钥 | 密钥对（公钥+私钥） |
| **速度** | 快，适合大数据 | 慢，适合小数据 |
| **密钥管理** | 需安全共享 | 公钥公开，私钥保密 |
| **主要用途** | 数据加密 | 密钥交换、数字签名 |
| **典型算法** | AES、DES、3DES | RSA、ECC、DH |

### 7.3 数字签名与数字证书
**数字签名工作原理：**
1. 对文件进行Hash计算得到消息摘要
2. 使用私钥对摘要加密生成数字签名
3. 发送原始文件+数字签名
4. 接收方用公钥解密签名验证完整性

**数字证书作用：**
- 由CA签发，认证用户公钥
- 解决公钥真实性验证问题
- 遵循X.509体系标准

## 八、软件工程方法论（重点★★★★★）

### 8.1软件过程
- 描述：需求分析与规格说明
- 开发：设计与实现
- 有效性验证：测试、评审
- 进化：维护与升级

### 8.1 软件开发模型/软件过程模型
| 模型 | 特点 | 适用场景 |
|-----|------|---------|
| **瀑布模型** | 线性顺序，阶段明确 | 需求完全确定 |
| **原型模型** | 快速开发原型，迭代改进 | 需求难以完整定义 |
| **螺旋模型** | 快速原型+风险分析 | 庞大复杂高风险系统 |
| **V模型** | 测试与开发阶段对应 | 强调测试验证 |
| **RUP** | 用例驱动、架构中心、迭代增量 | 大型项目 |
| **敏捷方法** | 短迭代、轻文档、快速交付 | 规模小、需求变化快 |

### 8.2 敏捷开发核心
**敏捷宣言：**
- 个体和交互胜过过程和工具
- 可工作的软件胜过大量的文档
- 客户合作胜过合同谈判
- 响应变化胜过遵循计划

**Scrum框架：**
- 角色：产品负责人、Scrum Master、开发团队
- 活动：Sprint规划、每日站会、Sprint评审、Sprint回顾
- 工件：产品待办列表、Sprint待办列表、增量成果

### 8.3 基于构件的软件工程(CBSE)
**构件特性（口诀：竹鼠挡毒标）：**
- 可组装性、可部署性、文档化、独立性、标准化

**构件组装方式：**
- 顺序组装、层次组装、叠加组装

## 九、面向对象分析与设计（超级重点★★★★★）

### 9.1 UML图分类
| 图类型 | 主要关注点 | 典型用途 |
|-------|-----------|---------|
| **用例图** | 系统外部用户与系统交互 | 捕获需求、确定系统边界 |
| **类图** | 类、属性、方法及类间关系 | 定义系统结构设计 |
| **顺序图** | 对象间消息的时间顺序 | 展示场景交互流程 |
| **通信图** | 对象间连接关系及消息传递 | 表达协作结构 |
| **活动图** | 工作流、活动顺序/分支/并行 | 表达业务流程 |
| **状态图** | 对象状态变化及触发条件 | 建模对象生命周期 |

### 9.2 类之间的关系（强度排序）
**泛化 = 实现 > 组合 > 聚合 > 关联 > 依赖**

| 关系类型 | 定义 | 图例特点 |
|---------|------|---------|
| **泛化** | 父类与子类的关系 | 空心三角箭头指向父类 |
| **实现** | 接口与实现类的关系 | 虚线+空心三角 |
| **组合** | 整体与部分，生命周期相同 | 实心菱形 |
| **聚合** | 整体与部分，生命周期可不同 | 空心菱形 |
| **关联** | 明确的语义联系 | 实线 |
| **依赖** | 一个类变化可能引起另一个变化 | 虚线箭头 |

### 9.3 面向对象设计原则
| 原则 | 描述 |
|-----|------|
| **单一职责原则(S)** | 类应该只有一个设计目的 |
| **开闭原则(O)** | 对扩展开放，对修改关闭 |
| **里氏替换原则(L)** | 子类可以替换父类 |
| **接口隔离原则(I)** | 接口提供客户端需要的行为 |
| **依赖倒置原则(D)** | 针对接口编程 |
| **组合/聚合复用原则** | 通过组合或聚合复用，比继承更灵活 |
| **最少知识原则** | 软件实体尽量少与其他实体相互作用 |

## 十、层次架构设计（重点★★★★★）

### 10.1 表现层架构对比
| 维度 | MVC | MVP | MVVM |
|-----|-----|-----|------|
| **核心组件** | Model-View-Controller | Model-View-Presenter | Model-View-ViewModel |
| **数据流** | 单向 | 双向 | 双向绑定 |
| **视图访问模型** | 视图可直接访问 | 通过接口通信 | 数据绑定交互 |

### 10.2 数据访问模式
| 模式 | 核心思想 | 特点 |
|-----|---------|------|
| **在线访问模式** | 与数据库保持持续连接 | 实时性强，依赖网络稳定性 |
| **DAO** | 数据访问逻辑与业务逻辑分离 | 解耦数据层与业务层 |
| **DTO** | 跨进程/网络传输数据的轻量级容器 | 减少网络传输次数 |
| **离线数据模式** | 数据获取后本地存储 | 离线处理，支持XML/JSON |
| **ORM** | 对象与关系数据库映射 | 面向对象编程，自动生成SQL |

### 10.3 污水池反模式
- **问题**：请求流简单穿过各层次，层次间隔离失效
- **后果**：系统结构不清晰、可维护性降低
- **解决**：层次开放，数据处理直达核心层次

### 11.3 缓存问题与解决方案
| 问题 | 描述 | 解决方案 |
|-----|------|---------|
| **缓存穿透** | 查询不存在数据，直达数据库 | 布隆过滤器、缓存空值 |
| **缓存击穿** | 热点key过期，大量请求直达DB | 互斥锁、逻辑过期 |
| **缓存雪崩** | 大量key同时过期 | 随机过期时间、多级缓存 |
| **大Key问题** | 单个key数据量过大 | 拆分、压缩、清理 |
| **热Key问题** | 单个key访问过于集中 | 本地缓存、读写分离、拆分 |
